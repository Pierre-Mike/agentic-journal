## Intent

This blog is a laboratory for agentic engineering. It exists to prove, by continuous use, that the workflow I advocate actually works at small scale. If the workflow is wrong, the blog will break. If the workflow is right, the blog will feel effortless over time.

## Why

Most writing about AI agents focuses on the agent itself — better prompts, flashier tools, bigger models. That's the smallest lever. The real leverage is outside the agent: specifying intent, constraining execution, measuring outcomes, enforcing invariants deterministically.

To practice what I preach, every post here is an archived spec:

- A `proposal.md` declared the intent
- A `gate:` field pointed at this MDX file
- `scripts/tasks-verify.ts` confirmed the gate passed
- `scripts/spec-archive.ts` moved the spec from `active/` to `archive/` with a timestamp

You can read this post. You can also read the spec that produced it, at `specs/archive/2026-04-14-000-day-0-why-this-blog/`. Both are the source of truth — one for readers, one for the runtime.

## What

A few rules the blog operates under, spelled out so you can judge whether the workflow survives contact with reality:

1. **Spec-first.** No post, no feature, no config change ships without a spec. The human writes intent; the agent executes against it.
2. **One spec, one gate, one verdict.** Every spec declares how "done" will be verified. For posts, that's a writeup gate — required markdown sections must exist. For features, that's a test file. For architectural rules, that's a lint rule plus fixtures.
3. **Deterministic-first.** Anything a linter, hook, or test can enforce must not be a skill. LLM time is reserved for genuine judgement — the `align` skill at spec-authoring, and whatever remains genuinely ambiguous inside a task.
4. **State lives in the filesystem.** Active versus archive is a directory, not a frontmatter flag. `scripts/spec-status.ts` computes state; no agent decides it.
5. **The human is a gate, not a babysitter.** Approval lives upfront at spec time, not mid-execution. Traces and evals handle the rest.

If any of this breaks, the incident becomes the next post. That is the point.
