**`src/backend/expertise.yaml`**

```yaml
meta:
  domain: backend
  scope: domain
  updated: "2026-04-02"
  updated_by: "agent"

summary: |
  Backend services handling payments, webhooks, and data persistence.
  Currency is always stored as integer cents. Stripe webhook handling has
  a strict middleware ordering requirement that silently breaks if violated.

gotchas:
  - id: BE-G001
    title: "Stripe webhook validation must precede body parsing middleware"
    description: >
      Stripe signature verification requires access to the raw request body.
      If any body-parsing middleware (e.g. express.json()) runs first, the
      raw buffer is consumed and the signature check always fails — silently,
      with no helpful error. Register the Stripe webhook route before mounting
      any body parser, or use a route-specific raw body parser for that endpoint.

conventions:
  - id: BE-C001
    title: "Currency amounts are integer cents — never floats"
    rule: >
      All monetary values are stored and passed as integers representing cents
      (e.g. $12.99 → 1299). Never use floats for currency at any layer —
      storage, computation, or API responses. Convert to a display string only
      at the presentation layer.
```

---

**Why only two entries:**

- **BE-G001 (Stripe)** — passes all three gates: future agents would break this, it's invisible in code, and it's a stable architectural constraint.
- **BE-C001 (cents)** — same: a float lurking anywhere would cause real bugs, the convention isn't self-documenting, and it's a stable data model decision.
- **Repository pattern** — fails the non-obvious gate. Any agent can read two files and see the pattern. Saving that would be noise.
